# 로그인 방식 구성 가이드 (세션 vs JWT)

아직 방식을 고정하지 않았을 때, 이 저장소의 레이어(`routes` → `controller` → `service` → `repository`)에 맞춰 어디에 무엇을 두면 되는지만 정리합니다. 구현 코드는 포함하지 않습니다.

## 공통으로 두는 부분

- **자격 증명 검증**: `auth.repository`에서 사용자 조회, `auth.service`에서 비밀번호·상태 검증(해시 비교 등)을 합니다. 로그인 방식과 무관하게 동일합니다.
- **보호 라우트**: `src/middleware/`에 `requireAuth` 같은 미들웨어를 두고, `*.routes.ts`에서 필요한 경로에만 붙입니다. 미들웨어는 “이 요청이 인증된 사용자인가?”만 판별하고, 비즈니스 규칙은 controller/service에 둡니다.
- **환경 변수**: 쿠키 이름·도메인·시크릿·TTL 등은 `src/schemas/env.schema.ts`에 정의하고 `src/config/env.ts`에서 읽는 패턴을 유지합니다.

## 서버 세션 방식

**개념**: 인증 상태를 **서버 측 저장소**(메모리, Redis, DB 등)에 두고, 클라이언트에는 **세션 ID만** 담은 쿠키를 내려줍니다.

### 구성 포인트

| 영역 | 역할 |
|------|------|
| `app.ts` | `cookie-parser` 다음에 세션 미들웨어를 등록합니다. 순서가 바뀌면 세션 쿠키가 붙지 않을 수 있습니다. |
| `config/` | `secure`, `httpOnly`, `sameSite`, `maxAge`, 프로덕션에서의 `trust proxy` 여부 등 쿠키·세션 옵션을 모읍니다. |
| 저장소 | 개발은 기본 스토어로도 되지만, 프로덕션에서는 Redis 등 외부 스토어를 쓰는 것이 일반적입니다. |
| `auth.service` | 로그인 성공 시 `req.session`에 `userId`(또는 최소 식별자)만 넣고, 민감 정보나 큰 객체는 넣지 않습니다. |
| `auth.controller` | 로그인/로그아웃/`/me`에서 `req.session` 읽기·갱신·destroy만 orchestration 합니다. |

### 요청 흐름

1. `POST /login`: service에서 자격 증명 검증 → 성공 시 세션에 사용자 식별자 저장 → 응답과 함께 `Set-Cookie`(세션 ID).
2. `GET /me` 등: 세션 미들웨어가 쿠키로 저장소를 조회 → `requireAuth`가 세션에 식별자가 있는지 확인합니다.
3. `POST /logout`: `req.session.destroy()` 등으로 서버 측 세션을 제거하고 쿠키를 만료시킵니다.

### 프론트·CORS

- 브라우저가 요청에 쿠키를 포함하려면 `fetch`에 `credentials: 'include'`를 쓰고, API의 CORS는 `credentials: true`와 허용 origin을 **구체 도메인**으로 맞춥니다.
- 쿠키 기반이면 **CSRF**를 고려합니다. SameSite(`Lax`/`Strict`)만으로 부족한 경우(크로스 사이트 POST 등) CSRF 토큰·이중 쿠키 패턴을 검토합니다.

### 이 방식에 잘 맞는 경우

- 웹 앱과 API가 같은 제품 단위로 묶여 있고, **서버가 세션 저장소를 가질 수 있는** 배포 형태일 때입니다.
- 토큰 만료·폐기를 “서버에서 세션 삭제”로 단순화하고 싶을 때입니다.

## JWT 방식

**개념**: 서버는 **서명된 문자열(토큰)**을 발급하고, 이후 요청마다 토큰을 검증만 합니다. 액세스 토큰만 쓰면 **상태 비저장**에 가깝고, 리프레시까지 쓰면 **저장소(DB/Redis)**가 다시 필요해집니다.

### 구성 포인트

| 영역 | 역할 |
|------|------|
| `config/` / `schemas/` | 서명 키(`JWT_SECRET` 등), `issuer`/`audience`, 액세스·리프레시 TTL을 env로 관리합니다. |
| `auth.service` | 로그인 성공 시 `sign`(payload는 `sub`, `exp` 등 최소한)을 하고, 필요 시 리프레시 발급·로테이션 로직을 둡니다. 검증은 미들웨어 또는 작은 `auth/jwt` 유틸로 분리해도 됩니다. |
| 미들웨어 | `Authorization: Bearer <token>` 또는 **httpOnly 쿠키**에 둔 JWT를 읽어 `verify` 후 `req.user` 등에 식별자를 붙입니다. |

### 토큰을 어디에 둘지

- **JSON 본문**으로 `access_token`을 주고 클라이언트가 `Authorization`에 실어 보내는 방식은 구현이 단순하지만, XSS에 노출되면 그대로 탈취되므로, 웹에서는 **httpOnly 쿠키에 액세스 토큰**을 두는 선택지와 트레이드오프를 비교합니다.
- **액세스(짧은 TTL) + 리프레시(긴 TTL)**: 리프레시는 httpOnly 쿠키 + 서버에 jti/버전을 저장해 폐기·로테이션하는 패턴이 흔합니다.

### 요청 흐름

1. `POST /login`: service에서 검증 후 토큰(들) 발급 → JSON 또는 `Set-Cookie`.
2. 보호 라우트: 미들웨어에서 토큰 추출 → 서명·만료·필요 시 블랙리스트 확인 → 통과 시 다음 핸들러.
3. 로그아웃: 클라이언트에서 토큰 제거만으로는 서버가 모를 수 있으므로, **리프레시 폐기**(DB/Redis에 저장해 둔 jti 삭제)나 짧은 액세스 TTL에 의존하는지 정책을 정합니다.

### 이 방식에 잘 맞는 경우

- 모바일·서드파티 클라이언트가 **Bearer 헤더**로 붙이기 쉬워야 할 때입니다.
- API 서버를 **수평 확장**하고 세션 스티키/공유 저장소 없이 맞추고 싶을 때입니다(액세스 JWT만으로 한정할 때).

## 한 프로젝트에서 섞지 않기

- **한 클라이언트 타입당 한 가지 “주된” 인증 통로**를 두는 것이 디버깅·보안 정책에 유리합니다. 예: 브라우저는 세션, 모바일만 JWT 등으로 **명시적으로 분리**합니다.
- 결정 후 `back/README.md`의 컨벤션과 `AGENTS.md`에 “이 API는 세션 쿠키입니다 / Bearer입니다” 등으로 한 줄이라도 적어 두면 프론트·에이전트가 맞추기 쉽습니다.
